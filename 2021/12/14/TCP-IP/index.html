ICMP

是IP的补充，用于IP传输时的差错报告、拥塞控制、路径控制以及时间、掩码等信息的获取。

差错报告

丢弃出错的IP数据报

向信源机报告差错，不纠错，纠错留给高层协议

报文的数据区包含出错数据报的首部及其前64比特的数据，帮助信源机确定出错分组

报文作为一般数据传输

信宿不可达

当路由器无法转发数据报，或信宿机无法向上层提交数据时，丢弃当前的IP数据报，并向信源发送ICMP信宿不可达报告。

  1. 网络不可达 0
  2. 主机不可达 1
  3. 协议不可达 2
  4. 端口不可达 3

数据报超时

TTL超时：TTL减为0

分片重组超时：信宿机分片重组时会启动定时器

数据报参数错

路由器或信宿机发现IP首部的字段值有问题或空缺，就丢弃该数据报，并向信源发送ICMP数据报参数错报告。

控制报文

拥塞控制

源抑制控制报文

为什么要源抑制：IP时无连接的，中间路由不会预留缓冲区。当大量IP数据报进入路由器或信宿机时，可以能会造成其缓冲区溢出，出现拥塞。

源抑制步骤

发现拥塞：路由器或者信宿机如果因为缓冲区溢出而丢弃数据报，立即向丢弃数据报的信源发送ICMP源抑制报文

解决拥塞：信源机按一定的规则，降低去往相应信宿的数据报速率，直到不再收到更多的源抑制报文为止

恢复：如果信源机在一段时间没有收到相应源抑制报文，则认为此方向拥塞已解除，则逐渐恢复去往相应信宿的数据报发送速率。

路径控制

重定向报文：用于主机路由表的更新，实现路径控制。

路由器和主机都有路由表(可使用route print或netstat –r 命令查看) 。路由器的初始路由表只含直接相连网络或管理员设置的路由表；主机的初始路由表只包含默认网关。

路由表用ICMP重定向报文进行更新

请求应答

回应请求与应答报文

作用：用来测试设备的网络可达性，以及设备IP协议是否正常工作。
ping命令：基于ICMP回应请求与应答报文。

时间戳请求与应答报文

作用：网络设备的存在时钟差异，而一些分布式应用要求各设备的时钟是同步的。此类报文可实现设备的时钟同步。
基本思路：设备通过此类报文，获取信息往返另一设备的时间戳信息，从而估算出设备间时钟差异，并进行相应调整。

地址掩码请求与应答报文

作用：获取设备所在网络的掩码。
应用：无盘机通过RARP获得IP地址后，可向所在网络的路由器发送ICMP地址掩码请求报文，获得所在网络掩码。

路由器请求与通告报文

作用：获取所在网络的默认路由器的IP地址。
实现过程：主机引导时会广播或组播路由器请求报文，所在网络的路由器收到后返回通告报文，通告本网络所有路由器地址和优先级，主机选择优先级高的作为其默认路由器。

IP路由

直接传递：在信宿网络中，直接传到信宿

间接传递：在非信宿网络中的传递过程

网关：网间连接器，在传输层上以实现网络互连，用于两个高层协议不同的网络互连

路由表

信宿地址一般采用信宿网络地址

网络远少于主机，减小路由表
与网络拓扑结构相对应。网络拓扑中，只有路由器和网络，没有主机
主机的增减不会对路由表产生影响
减小使用和维护的开销，简化设备的设计和实现

路由表字段

信宿地址：一般为信宿网络地址。特殊情况：特定主机路由表项中为信宿主机地址;在默认路由表项中为0.0.0.0

子网掩码：一般为信宿网络的掩码。在特定主机路由表项中为255.255.255.255; 默认路由表项中为0.0.0.0。

下一跳地址：间接交付：下一个路由器的入口地址;直接交付：
本路由器在信宿网络的出口地址

输出接口：指明数据报在本设备输出的接口。

度量：本设备到信宿机的距离，一般为跨越的路由器的个数，路由器到直连网络的跳数为1。

路由表表项

普通路由表项

信宿网络地址 + 信宿网络的掩码

特定主机路由表项

信宿主机地址 + 255.255.255.255
作用：为特定主机指定路由，用于网络测试或安全等

默认路由表项

信宿地址0.0.0.0 + 掩码0.0.0.0
一般位于路由表的最后
作用：为剩余目的网络指定路由，可减小路由表，隐藏网络细节

表项的顺序：直接交付表项 > 特定主机路由表项 > 普通路由表项 > 默认路由表项

动态路由

路由器之间周期性地交互路由信息，根据最短路径优先原则，计算并更新各自的路由表。
最短路径优先原则特点：只考虑路径长度，不考虑拥塞和负载等，不保证所选路径是最优的。计算简单，可以简化路由协议的设计和实现。

VD vector-distance

路由表更新策略

假设A收到B的路由表

对于目的网络N，A的下一跳是B，则按照B中的N更新

新项目直接加进来

下一跳不是B，并且距离更短

缺点：

收敛慢

LS link status

收敛速度较快：链路状态信息的发布是面向所有路由器，每个路由器都能够迅速获知变化。
信息交换量较小：只发布自己与相邻路由器的链路状态信息；此后只在网络变化时，发布变化部分。
扩展能力强：可采用分层的结构。

缺点：

路由器计算量大

路由信息协议RIP

路由环路：

由于V-D算法收敛慢，可能由于信息不一致导致路由环路。因此，RIP规定：有效路径的跳数不能超过15，跳数为16为无效路径。

相同距离路径问题
若去往某一信宿存在多条距离相同的路径，先入为主

失效路径

如果在6个周期(3分钟)内，没有收到与某一表项相关的V-D信息，就将该表项的度量值设置成16，并标注为删除。等过一段时间，该路径失效信息传播开去后，才真正删除该表项。

计数到无穷

慢收敛的极端情况

水平分割法：

如果路由器从某个接口收到某些网络的VD信息，不允许再从此接口发出这些网络的信息。

毒性逆转法：

从I接口收到VD信息，这个接口发送此网络的VD信息跳数设置为16（水平分割法的变形）

保持法：

在直连网络不可达的一段时间之内，禁止用收到的VD报文刷新此不可达信息

RIP报文及封装

帧首部 IP首部 UDP首部 RIP报文 帧尾部

v2改进：

支持子网，支持可变长子网掩码和无类别域间路由CIDR
提供报文鉴别机制，采用口令方式
采用组播来发送路由通告，组播地址224.0.0.9对应所有RIP v2路由器

OSPF

每个区域至少有一个区域边界路由器ABR用于汇总本区域的路径信息

OSPF采用链路-状态算法，每个路由器测试自己与其相邻路由器的链路状态信息，发现变化后在区域内以泛洪方式广播变化信息。
每个路由器(包括ABR)收到区域内路由器的信息后，可得知区域的网络结构，计算最新最短路径优先树，并进行相应路由表更新。而区域之间的路由信息则通过ABR进行交互。

采用层次结构，支持扩展，易于管理。
支持特定主机、特定子网、分类网络、无类网络的路由。
不同的区域可以使用不同的报文鉴别方法。
采用组播，减少不相关设备的负担。

BGP

自治系统边界路由器(ASBR)通过内部网关协议(例如RIP或OSPF)收集自治系统内部的路由信息，然后与其他自治系统的ASBR通过BGP交换信息（信宿，下一跳和去往信宿的路径）。
相邻的ASBR收到后，计算并刷新自己的路由表。

BGP的路由不是最短路径优先，而是基于管理员配置的路由策略，通常与政治、经济或安全等有关。
BGP协议位于应用层，使用TCP作为传输层协议。两个ASBR路由器在交换路由信息前，必须先建立一条TCP连接。TCP协议将在下一章介绍。

传输层

三元组标识一个进程：

主机地址，传输层协议，端口号

网络通信是使用相同传输层协议的两个进程的通信。

五元组标识一个网络通信:

传输层协议, 本地主机地址, 本地进程端口号, 远方主机地址, 远方进程端口号

进程通信常采用客户机/服务器(C/S)方式：

熟知端口：

由服务器进程使用，有公认用途的端口，包含TCP和UDP的0-1023端口：
SMTP服务器进程：TCP 25端口
DHCP服务器进程：UDP 67端口
FTP服务器的控制进程：TCP 21端口
FTP服务器的数据进程：TCP 20端口

临时端口：由客户进程使用，通信前向操作系统申请，通信后归还，包含TCP和UDP的1024~65535端口。

TCP

传输控制协议TCP (Transmission Control Protocol)：面向连接的传输层协议
特点：基于无连接、不可靠的IP，采取了一系列措施，例如建立和拆除逻辑连接，流量控制、拥塞控制、差错控制等，向应用层提供面向连接的、可靠的字节流传输服务。

TCP段的格式

将应用层要传输的字节流分块，封装成若干个TCP段:
段首部：固定首部（20个字节） + 选项（0到40字节之间，且必须为4字节的整数倍）
数据：2字节的整数倍

TCP总长度=IP中数据报总长度字段-首部长度*4

求校验和：首先，把伪首部、TCP报头、TCP数据分为16位的字，如果总长度为奇数个字节，则在最后增添一个位都为0的字节。把TCP报头中的校验和字段置为0

其次，用反码相加法累加所有的16位字（进位也要累加）。
最后，对计算结果取反，作为TCP的校验和。

序号

TCP连接中传送的数据流中的每一个字节都编上一个序号。
序号字段的值则指的是本报文段所发送的数据的第一个字节的序号。
如：流序号为=X，数据长度=L
则：报文的序号为X，下一个 报文序号为X+L
特点
报文的顺序关系
数据流的位置，更便于流的复原
较大的序号空间（32bits，4Gbyte）
报文的序号不连续

TCP选项

结束选项：作为整个TCP选项的结束。

最大段大小(MSS)选项：4字节。内容字段告诉对方，本机所在网络能够接受的最大TCP段数据区的字节数。如以太网的MSS可达1460字节，默认为536字节。

固定首部中的窗口字段最多只能表示65535个字节的空闲接收缓存大小。本选项用来扩大其表示，扩展后的空闲缓存大小为：
Wn = Wo×2f
Wo为窗口字段的值，f为窗口规模因子。

连接的建立需要请求和响应，但是基于不可靠IP，可能丢失。
TCP：使用超时重传来解决IP不可靠问题。
重传导致重复连接问题。
TCP：根据TCP段的序号来查重。
为了可靠建立连接，TCP设计了三次握手方式。

三次握手

SYN seq = x

ACK SYN seq = y ack = x+1

ACK seq = x+1 ack = y+1

四次挥手

->FIN ACK seq = u ack = v

<-ACK seq = v ack = u+1

<-FIN ACK seq = w ack = u+1

->ACK = 1 seq = u+1 ack = w+1

TCP流量控制

控制发送方发送数据的速率，使得接收方来得及接收和处理。

TCP拥塞控制

拥塞：网络中传输的数据过多，造成路由器缓存溢出导致数据丢失。数据丢失引起超时重传，进一步加剧网络拥塞。

TCP拥塞控制：也是控制发送方的发送速率，使得网络来得及传输。

TCP流量控制：发送方的发送窗口（swnd）不能大于接收方的接收窗口（rwnd）

拥塞窗口(cwnd)：表示当前网络的转发能力，由发送方根据网络状况动态设置。

TCP差错控制

段丢失。检测：发送方使用重传计时器。处理：超时重传。确认段的丢失不需处理，因为TCP采用累计确认。
段损坏。检测：接收方使用校验和字段。处理：丢弃出错的段，不返回确认段，由发送方超时重传。
段重复。一般由超时重传造成的。检测：接收方分析段序号。处理：直接丢弃重复段。
段失序。由于TCP基于IP，不能保证按序到达。处理：接收方对先到达的段暂不确认，放入接收缓存，直到前面的段都到齐了再一起向上提交并返回确认。

重传计时器
发送方发送每个段时，启用重传计时器，如在超时时间内未收到对方确认，则重传。
超时值：最常用的：
$$
Timeout = 2\cross RTT
$$

$$
RTT=\alpha \cross RTT_0+(1-\alpha)\cross RTT_1
$$

其中RTT0为之前计算的RTT值，RTT1为最新计算的RTT值，系数α通常为0.9。
RTT0和RTT1的计算：

  1. 通过TCP段的时间戳选项；
  2. 通过计时器。

UDP

DNS

Domain Name System

DNS既可以基于TCP，也可以基于UDP，服务器则使用熟知端口53
对UDP或TCP的使用原则：

使用A查询请求某个域名对应的IP地址时使用UDP。

如果响应报文长度大于512字节，则UDP仅返回前512字节，并设置报文首部“参数”字段的“截断”位。客户收到这个响应后，使用TCP重新发送原来的请求。

如果一次查询的名字很多，则客户可能会直接使用TCP。

在主域名服务器和辅助域名服务器间进行区域传送时，使用TCP。
核心：数据传输量少，可靠性要求低的时候用UDP；否则用TCP。

引导协议与动态主机配置

计算机上网通常需要：
IP地址
所在网络的掩码
默认路由器的IP地址
默认DNS服务器的IP地址

BOOTP：

BOOTstrap Protocol

位于应用层，基于UDP协议
作用：使计算机获得上述四种网络参数。
RARP使用链路层广播也能获取IP地址，但不能跨路由器。因此，RARP很少用了。

服务器进程使用UDP 67端口侦听。
客户机进程使用UDP 68端口在网络层中广播BOOTP请求(目的地址255.255.255.255，目的端口UDP 67；如果不知道自己的IP地址，则源地址0.0.0.0，源端口UDP 68)，数据中包含客户机的MAC地址。

DHCP

动态主机配置协议

客户机启动时，广播DHCPDISCOVER报文。
服务器响应DHCPOFFER报文(单播)，提供了可用于客户机的IP地址等信息。
客户机一般选择最先到达的响应，并向相应服务器发送DHCPREQUEST报文(广播)。
相应服务器收到后，返回DHCPACK报文。客户机收到后，开始租用此服务器提供的IP地址(单播) 。

反正客户机始终都是单播

1）如果收到DHCPDISCOVER报文，则从地址池中选择一个空闲IP，并提供租用时间，构造DHCPOFFER响应报文；同时把此IP地址锁定，并记录于地址分配表。
2）如果收到DHCPREQUEST，根据客户机的硬件地址，查找地址分配表，若找到，则响应DHCPACK报文；否则响应DHCPNAK报文。
3）如果收到DHCPDECLINE报文，会禁用报文中的IP地址，不再给客户机分配这个IP地址。
4）如果收到DHCPRELEASE报文，则解除地址绑定，此IP地址可被重新分配。

兼容BOOTP，BOOTP客户机可向DHCP服务器请求静态IP配置。报文格式基本相同；服务器进程UDP 67端口，客户进程UDP 68端口；
为特定客户机分配固定的静态IP地址；
为普通客户机分配临时的动态IP地址；
客户机可对多个服务器提供的服务作出选择和响应；
可以跨网通信（通过路由器的中继代理程序），不需要在每个网络上都设置服务器。

FTP

位置：应用层，基于传输层的TCP。
作用：实现不同计算机之间可靠的文件传输。不同操作系统、不同字符集、不同文件结构和存储方式等。

FTP建立两条TCP连接：
一条传送控制信息(命令和响应) ；
另一条传送文件数据。
双方4个进程：
服务器控制进程：TCP 21端口
服务器数据进程：TCP 20端口

SMTP

(1)建立连接
客户进程周期性扫描邮件缓存。如发现有邮件要发送，就主动发起并建立与服务器进程的TCP连接。
TCP连接建立完毕，服务器返回“220 Service ready”
客户机向服务器发送HELO命令，附上自己的域名，用来表明自己是谁。
服务器有能力接收邮件，则返回“250 OK”，否则返回“421 Service not available”。

(2)邮件传送
客户机发出MAIL FROM命令，后跟发信人地址。如：MAIL FROM 123@163.com
若服务器已准备好接收，则返回“250 OK”；
否则，返回具体错误代码。如：451(处理时出错)，452(存储空间不够)，500(命令无法识别)等。
收到“250 OK”后，客户机发出RCPT TO命令，后跟收信人地址
例如： RCPT TO 456@mail.njust.edu.cn

(3)拆除连接
客户机发送QUIT命令，表示邮件发送完毕。
服务器返回“250 OK”。然后双方拆除TCP连接

Telnet

位置：工作在应用层，基于传输层的TCP，服务器进程使用TCP 23端口。
作用：提供远程登录功能，客户机将命令传送到服务器上执行，服务器返回执行结果。

NVT的作用：建立数据表示的一致性。
Telnet客户端：用户的击键信息发送给终端驱动程序，这些本地字符被送到Telnet客户进程转换成NVT ASCII字符，再发送出去。

三角路由问题

三角形路径问题会引入流量的时延，容易引起归属代理处的路由瓶颈。例如，即使在校园网这样的网络规模中，移动IP的三角路由也会使报文的平均时延增加45%。

解决办法：将移动主机的转交地址与其归属地址进行绑定，当归属代理收到发给移动主机的第一个包时，将包转发给外区代理，同时发一个更新绑定包给远地主机，以便远地主机以后向这台移动主机发送信息时直接使用转交地址。远地主机将绑定信息存放在高速缓冲中。

出现新问题：移动主机的再次移动会造成远地主机高速缓冲中内容的过时。所以当移动主机再次移动时，归属代理必须向远程主机发送报警包，通告变化。
